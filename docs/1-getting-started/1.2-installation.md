# 1.2 安装与启动

本节说明 RPG Agent MV 的运行前提、启动方式和基础自检流程。

## 运行前提

使用前应准备以下条件：

- 一个数据和核心脚本完整的 RPG Maker MV 或 MZ 工程目录。MZ 1.10.0 是完整验证基线；旧版核心和加密资源会在导入时显示兼容警告，加密素材相关能力可能受限。
- 可用的模型供应商、API Key 和默认模型。
- RPG Agent MV 可读取的本地项目路径。
- 建议将 RMMV 项目纳入 Git 工作树，以便对比和整理改动。
- 编辑只需要 MV/MZ 源工程。顶部 Play 会优先使用工程自带运行器，其次使用已保存或官方安装位置中的完整运行库；仍找不到时会让你选择一次同引擎的完整 Windows 运行库目录并记住。应用不会扫描任意游戏目录，也不会联网下载运行器。

## 获取与启动

RPG Agent MV 是本地桌面应用。使用安装包或部署包时，应以对应版本的发布说明为准。不要使用来源不明的压缩包、安装器，也不要在非官方页面提交 API Key 或本地凭证。

从源码启动需要 Node.js 22.5 或更高版本。

源码依赖安装不会构建本地 Agent 运行工具：

```powershell
cd RPG-Agent-MV
npm run install:deps
```

如果你要在源码模式下直接使用本地 Agent，需要在开发机或发布构建机上显式构建一次运行工具：

```powershell
npm run build:opencode-runtime
```

公开安装包会把这些运行工具一起打进去，用户安装和启动应用时不需要自己构建。

## 应用内更新

使用经过签名的 **NSIS 安装版** 时，应用只从 GitHub Releases（[Lunaroa/RPG-Agent-MV](https://github.com/Lunaroa/RPG-Agent-MV)）检查更新，不使用备用镜像或自定义更新地址。

发现新版本后会提示；确认后下载并在用户同意重启时完成升级。客户端会尽量使用 blockmap 做差分下载；无法差分时由更新库改为下载完整安装包。源码开发模式（`npm run dev`）不会检查更新，也可以在设置 → 界面中手动「检查更新」。

安装包必须包含发布者签名信息。未签名、更新配置缺失或更新仓库不匹配的安装版会拒绝联网检查，不会降级到不安全的更新方式。

### 维护者发版

1. 将产品根与 desktop/backend 的 `version` 更新为同一版本。
2. 配置 Windows 代码签名证书，以及 `GH_TOKEN` 或 `GITHUB_TOKEN`。
3. 确保 Git 工作区干净后执行发布：

```powershell
cd RPG-Agent-MV
npm run release:electron:publish
```

发布命令会先检查工作区、版本和令牌，再强制生成签名安装包并运行发布检查；任何一步失败都会停止，不会上传部分结果。验收 GitHub Release 中包含 `latest.yml`、安装包 `.exe` 和对应 `.blockmap`。

## 启动后自检

首次启动后建议依次确认：

1. 左侧导航可进入编辑器和控制台。
2. 控制台可显示当前项目。
3. 编辑器可加载至少一张地图。
4. 设置中存在可用模型。
5. Agent 面板可以发送简单请求。

上述任一项失败时，应先完成环境或项目排查，再执行复杂事件任务。

## 常见启动问题

| 现象 | 处理建议 |
|------|----------|
| 项目未显示 | 确认目录包含可读取的 `data/System.json`、`MapInfos.json`、地图和对应核心脚本；工程标记不是必需文件 |
| MV/MZ 源码工程无法试玩 | 按提示选择同引擎、包含 `Game.exe` 和完整 NW.js 文件的 Windows 运行库目录；MZ 1.10.0 是完整验证基线，其他可识别 MZ 版本允许尝试启动但不承诺完整兼容；选择结果会保存在本机，应用不会扫描私人游戏目录或联网下载运行器 |
| 模型不可用 | 检查供应商、API Key、Base URL 和默认模型，详见 [5.3 模型与运行检测](../5-faq/5.3-model-check.md) |
| 地图无法打开 | 检查地图文件、图块集和本地素材是否完整 |
| 应用反复闪退 | 关闭应用并确认后端进程已退出，再重新启动 |
