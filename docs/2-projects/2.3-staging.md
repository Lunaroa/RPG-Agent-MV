# 2.3 暂存、应用与回退

本节说明 RPG Agent MV 的暂存机制，以及如何查看、应用或丢弃本轮改动。

## 暂存机制

桌面编辑器或 Agent 修改地图、事件、数据库、插件和素材时，先在产品运行目录中创建草稿，并记录源文件哈希。保存到暂存不会改写源工程；只有通过应用预检后才会写回。

Agent 数据库操作还会记录操作归属、会话、计划哈希和占用文件。数据库、系统、类型和术语等共享文件不能被多个操作同时占用。旧暂存仍作为用户草稿处理，不会被自动迁移成 Agent 操作。

## 查看本次改动

应用前建议检查以下信息：

- Agent 在聊天流中的改动说明。
- Inspector 中可折叠的字段差异、受影响文件和校验错误。
- 编辑器中地图或事件的实际变化。
- 项目位于 Git 工作树时的 `git diff`。
- 运行日志中的阻断错误或风险提示。

## 应用改动

满足以下条件后再应用改动：

- 改动范围符合任务目标。
- 地图和事件位置正确。
- 没有阻断性错误。
- 没有无关文件或无关事件改动。

Agent 数据库操作必须通过绑定当前会话、操作编号和计划哈希的审批。用户在桌面端亲自点击“应用全部”时可以一并应用，但界面会列出包含的 Agent 操作，并执行相同的源文件、草稿和语义预检。多文件修改要么全部成功，要么全部不写。

## 丢弃改动

以下情况应丢弃本轮改动：

- 写入了错误地图或错误事件。
- 事件内容方向明显不符合需求。
- 写入范围越界。
- 存在未解决的阻断问题。
- 用户决定不保留本轮尝试。

丢弃会移除草稿，源文件保持不变。数据库或公共事件还可在 Inspector 中仅“撤销此条暂存修改”，不影响同一文件中的其他条目；当文件完全恢复后，对应暂存项会自动移除。

## 暂存与保存版本

| 方式 | 作用 | 适用场景 |
|------|------|----------|
| **丢弃暂存** | 将本轮未保留改动还原到快照 | 当前任务结果不符合预期 |
| **保存版本** | 把已写入工程的改动记入本地版本历史（可写版本说明） | 需要留下带说明的里程碑，便于日后对比或回退 |

保存版本在控制台右上角 **版本管理** 区域操作，只需本机安装 Git，**不需要远程仓库**。启用版本管理后即可随时保存；若当前没有新改动，会提示「没有新的改动需要保存」。

## 冲突处理

应用暂存时如出现冲突，通常说明工程文件在暂存之后被外部工具修改、草稿被改写，或文件被其他操作占用。系统会直接阻止应用，不自动合并或修补。确认 RPG Maker MV 或其他工具是否改写了项目文件后，再决定重新读取、丢弃暂存或人工处理。

## 试玩与暂存

顶部 **Play** 使用当前工程对应的运行器启动源工程，从 `System.json` 配置的起点开始。它先检查工程自带运行器；未部署 MZ 工程使用 `nwjs-win\nw.exe`，未部署 MV 工程使用原版试玩目录中的 `nwjs-win-test\Game.exe`。右键 Play 可以查看或切换当前引擎的运行器路径。运行器不会被复制或下载。MZ 1.10.0 仍是完整验证基线，其他可识别版本只允许尝试普通试玩。互动试玩不包含暂存；存在暂存时必须先确认当前暂存摘要，摘要变化后需要重新确认。运行游戏可能写入工程的 `save/`。

Agent 自动验证使用系统临时目录中的隔离副本：复制源工程内容但排除存档和 MZ 运行器、覆盖当前全部暂存，再由源工程中校验通过的 `Game.exe` 启动该临时副本。验证会在有限时间内检查截图、实际地图坐标、运行时就绪、事件空闲、JavaScript 错误以及源工程/存档是否未变。证据不足会返回阻断、需复核或失败，不会降级成“通过”。互动试玩卡片只证明进程生命周期，不证明剧情或可玩性已经通过。
