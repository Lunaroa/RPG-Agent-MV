# 3.1 需求描述与决策

清晰的需求描述可以减少错误生成和无效修改。本节说明如何描述事件任务，以及如何处理执行过程中的 ASK 决策。

## 需求描述原则

模糊需求会导致结果不可控。一个有效需求通常包含：

- 目标地图或已有事件对象。
- 要生成或修改的内容。
- 剧情意图、台词方向或任务目标。
- 不允许修改的范围。
- 新事件是否只生成待放置版本。
- 写入或放置前的决策规则。

## 示例

不推荐的描述：

```text
增加一段剧情。
```

推荐的描述：

```text
为 Map003 准备一个待放置的告示牌事件。玩家调查后显示：村长今晚在旅馆等你。
不要改主线开关，不要直接决定坐标；事件进入待放置队列后，由用户在编辑器中拖放位置。
```

## ASK 决策

当任务涉及地图选择、剧情取舍、素材选用、冲突处理或结果接受等高风险节点时，系统会通过 ASK 暂停并等待明确决策。

ASK 是执行流程中的决策机制，但不是所有后端写入 API 的硬性锁。即使未看到 ASK，也应在改动说明、编辑器和日志中检查本轮任务的实际影响范围。

## 回答方式

回答 ASK 时应给出明确约束。例如：

```text
用 Map003，做成待放置事件；位置由用户拖放到村口附近，不要修改已有传送事件。
```

或：

```text
保留村长隐瞒真相的方向，但语气不要过于阴沉，先做成轻微不安。
```

避免使用缺少目标、范围和边界的模糊表达。决策信息越明确，后续检查越容易判断是否越界。
