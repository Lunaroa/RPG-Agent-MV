# 3.4 动态工作流

本节说明动态工作流从提议、审批到执行报告的完整流程。

## 适用场景

当一个任务需要并行扇出多个只读子 Agent（例如同时清点多张地图的事件、交叉核对多个素材引用、分头读取上下文再汇总），主 Agent 可以提交一份**编排脚本**，由编排引擎统一派发，而不是手动逐个调用子 Agent。

动态工作流不是必经流程；简单的单步任务仍由主 Agent 直接完成。

## 流程

1. **提议**：主 Agent 调用工作流提议工具（`workflow.propose`）提交编排脚本。脚本持久化保存，供用户审查。
2. **审批**：桌面端在 Agent 调用工具处暂停，弹出风险审批卡片，展示待执行脚本。用户选择：
   - **允许一次**：仅本次执行；
   - **始终允许**：后续工作流提议自动批准；
   - **拒绝**：放弃本次工作流。
3. **执行**：批准后，编排引擎按脚本里的 `agent()` / `parallel()` / `pipeline()` 原语派发只读子 Agent，自带 schema 校验和并发限制。子 Agent 的启动与结束实时显示在子 Agent 面板。
4. **报告**：工作流完成后，执行报告作为闭环串入会话流，供用户核对结果。

## 边界

- 工作流提议仅对**非只读**主 Agent 开放；只读子 Agent 不能再提议工作流。
- 工作流期间会话不会提前结束；中止会话会取消未完成的工作流。
- 工作流产物（运行记录、报告）落在产品运行时目录，不在 RMMV 工程内写入。
