# RPG Maker MZ 1.10.0 Official Compatibility Evidence

This page maps official RPG Maker requirements to implementation areas and focused tests within RPG Agent MV's supported scope. Specification evidence is limited to official RPG Maker sites, help pages, and script references. Private user projects are used only through local temporary-copy acceptance and are never placed in documentation, tests, fixtures, logs, or Git.

## Fixed Scope

- MZ project directories with complete readable data, core scripts, and a recognizable core version are accepted without requiring `game.rmmzproject`. MZ 1.10.0 is the fully validated baseline; older cores and encrypted resources require compatibility confirmation.
- Maps use the official **Auto plus Layers 1–4**, together with the format's shadow and region planes. Infinite layers and private map formats are not added.
- Character generation, dungeon generation, DLC, tutorials, deployment tooling, and an MZ-editor replica are outside scope.
- Editing does not require an MZ installation. Normal playtest prefers the statically validated runtime beside the project's `Game.exe`, then a user-selected `nwjs-win/nw.exe` from a local MZ installation. Battle Test and particle preview retain their stricter project-local runtime boundary.

## When Official Pages Disagree

The general database help still shows older 2000/1000 limits, while the official feature page explicitly names Actors, Classes, States, Skills, Items, Weapons, Armors, Enemies, Troops, and Common Events as supporting 9999. RPG Agent MV applies 9999 only to those ten groups and does not extrapolate it to Animations or Tilesets.

If an official page or PDF conflicts with the actual 1.10.0 core format elsewhere, the affected write must stop and gain a reproducible test. The product must not guess, silently downgrade, or rewrite core scripts.

## Product-Owned Safety Rules

These are RPG Agent MV rules, not claims about the MZ specification: every edit enters staging first; users confirm placement for new events; legacy MV sessions and tool names remain compatible; and private projects must never enter tests, fixtures, documentation, logs, or Git.
